# Architecture — knovas-extract system

How the cross-language extraction library family is structured, why we made the
choices we made, and where to look when adding to it.

If you only want to **use** an extraction library, see [`quickstart.md`](quickstart.md)
instead.

## 30-second mental model

```
                       ┌────────────────────────────────┐
                       │  knovas-extract-spec  (THIS)   │
                       │                                │
                       │  schema.json   ← contract      │
                       │  MANIFEST.yaml ← version/hashes│
                       │  corpus/       ← {input, JSON} │
                       │  docs/         ← semantics     │
                       │  tooling/      ← validators    │
                       └────────┬───────────────────────┘
                                │ pinned by spec_version
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
   ┌────────────────────┐ ┌──────────────┐ ┌──────────────┐
   │ knovas-extract-    │ │ knovas-      │ │ knovas-      │
   │   python           │ │   extract-   │ │   extract-   │
   │ (shipped 0.1.0)    │ │   node       │ │   go ...     │
   │                    │ │   (future)   │ │   (future)   │
   │ PyMuPDF/python-docx│ │ pdfjs+mammoth│ │ unidoc...    │
   └────────────────────┘ └──────────────┘ └──────────────┘
```

The spec is the source of truth. A language implementation is **correct** iff
its output matches the golden corpus at the spec version it pins, and its
adversarial behavior matches what the MANIFEST declares.

## Why spec-first?

A client asked for extraction libraries in many languages: Python first, then
Node, Go, Rust, Java. We considered three architectures:

| Architecture | What it costs |
|---|---|
| Single core (Rust + FFI / Tika + sidecar) | One implementation, but FFI marshaling overhead, JVM dependency, and Rust's PDF/DOCX/MSG ecosystem is thin — we'd wrap C libs (mupdf, libzip) anyway. |
| Per-language WASM core | Write-once, run-anywhere, but PDF rendering in WASM is single-threaded with no streaming, and the bundle bloats fast (20–50 MB once mupdf/libzip are embedded). |
| **Per-language native, governed by a shared spec ← chosen** | Each language uses its **best** native libs (PyMuPDF in Python, pdfjs in Node, Tika in Java, …). Cross-language consistency enforced by a versioned spec + corpus. |

"Spec-first" means: the spec ships first, fixtures ship second, language
implementations validate against them. A `knovas-extract-node` PR is "correct"
iff it passes `spec_version 1.0.0` corpus tests within the documented
[tolerances](tolerances.md). No language drifts silently.

## The contract

[`schema.json`](../schema.json) — JSON Schema 2020-12 for `ExtractionResult`:

```jsonc
{
  "spec_version": "1.0.0",
  "source": {            // computed by extractor, not parsed
    "mime_type": "...",
    "sha256": "...",
    "size_bytes": 12345,
    "filename": "..."    // nullable
  },
  "metadata": {
    "title": null, "author": null, "language": null,
    "created": null, "modified": null,
    "page_count": null, "word_count": null,
    "extra": {           // format-specific, namespaced (pdf:*, docx:*, …)
      "pdf:producer": "..."
    }
  },
  "content": {
    "text": "...",       // NFC + LF + trimmed; cross-language equality basis
    "pages": null,       // [{ index, text }] when paginated
    "sections": null     // [{ heading, level, text }] when headed
  },
  "warnings": [],
  "extractor": { "name": "knovas-extract-python", "version": "0.1.0" }
}
```

Field-by-field semantics in [`schema-fields.md`](schema-fields.md). What counts
as "equal" across languages in [`tolerances.md`](tolerances.md).

## The two corpora

### Golden — `corpus/<format>/`

Pairs of `(input, expected.json)`. A language implementation passes iff
`extract(input)` produces output equal to `expected.json` within the
[tolerances](tolerances.md) (default: text Levenshtein 0.5%, dates ±1s).

Golden fixtures are **synthetic and deterministic** — generated by
[`tooling/generate_synthetic_fixtures.py`](../tooling/generate_synthetic_fixtures.py)
with pinned timestamps + ZIP_DEFLATED so the SHA-256 stays byte-identical
across maintainer machines.

Today (`corpus_version 2026.06.25`): 9 fixtures across pdf, docx, txt, md,
html, rtf, eml.

### Adversarial — `corpus/adversarial/<format>/`

**Deliberately malicious inputs.** Don't open in viewers outside a sandbox.
Each fixture's `MANIFEST.yaml::adversarial[*].expected` declares the
behavioral contract — typed error class, "warning" (extracted but flagged),
or "sanitized" (extracted and the payload safely dropped).

Today: 5 adversarial fixtures covering encrypted PDF, zip-slip DOCX,
decompression-bomb DOCX, billion-laughs HTML, RTF object-linking (CVE-2017-0199
class).

Every adversarial fixture must carry a `provenance` field — synthesized, public
CVE PoC reference, etc. Validation refuses entries without it.

## Versioning

- **`spec_version`** — `major.minor.patch`. Bumps when `schema.json` changes.
  Major = breaking; every language must update before claiming conformance.
- **`corpus_version`** — calver `YYYY.MM.DD`. Bumps when fixtures change.
  Non-breaking by definition (a new fixture only adds a new gate).

A language implementation's version is independent. `knovas-extract-python
0.1.0` happens to pin `spec_version 1.0.0` and `corpus_version 2026.06.25`.

## CI flow

```
   PR / push                          PR / push
       │                                  │
       ▼                                  ▼
 ┌──────────────┐                  ┌──────────────────────────┐
 │ spec CI      │                  │ knovas-extract-python CI │
 │              │                  │                          │
 │ validate_    │                  │ lint + type              │
 │ corpus.py    │                  │ test (3×3 matrix)        │
 │              │                  │ security gates           │
 │ hash_fixtures│                  │ CodeQL                   │
 │ (idempotent) │                  │ CIFuzz (atheris)         │
 └──────────────┘                  │ secrets scan             │
                                   │   ↓                      │
                                   │   (golden tests pull     │
                                   │    spec via actions/     │
                                   │    checkout @ pinned ref)│
                                   └──────────────────────────┘
```

The python repo's `tests/golden/test_corpus.py` and `test_adversarial.py` are
parametrized at collection time from the spec corpus. Adding a fixture to the
spec automatically extends the test matrix in every language repo — no per-
language code change needed.

## Security posture (one paragraph)

`knovas-extract-*` is parser code running on untrusted input. The risks we
defend against — RCE via malformed PDFs (PyMuPDF wraps mupdf in C), XXE in
DOCX/HTML, decompression bombs in ZIP-based formats, RTF object linking,
header injection in email, supply-chain attacks via compromised PyPI deps —
are all enumerated in the per-repo `SECURITY.md`. Every release ships
**Sigstore-signed** wheels with **SLSA L3 provenance** and a **CycloneDX
SBOM**, publishable to PyPI via **OIDC Trusted Publishers** (no long-lived
tokens). For untrusted inputs, run extraction inside a sandbox; recipes
for `bubblewrap`/`nsjail`/rootless containers in the Python repo's
[`docs/sandboxing.md`](https://github.com/Seifeddini/knovas-extract-python/blob/main/docs/sandboxing.md).
